# DUH-RPC AI Cheat Sheet

Dense reference for AI agents building or modifying APIs that follow the DUH-RPC spec. For full rationale and examples, see [spec.md](spec.md).

---

## Endpoint Format

```
POST /v{version}/{problem-domain/}{subject}.{action}
```

- ALL requests use **POST**. No other HTTP verb.
- `problem-domain` is optional organizational namespace. Nesting is permitted.
- Subject comes before action (Yoda ordering): `/v1/dogs.feed` not `/v1/feed.dogs`
- Always include an action even for the primary operation: `/v1/kiss.give` not `/v1/kiss`

Examples:
```
POST /v1/users.create
POST /v1/users.list
POST /v1/billing/invoices.list
POST /v1/identity/sessions.create
```

CRUD mapping:
```
.create  — create a resource
.get     — read a resource (no side effects)
.update  — modify an existing resource
.delete  — remove a resource
.list    — paginated collection (see Pagination)
```

## Request Body

- Client MUST always send a body. For JSON: `{}`. For Protobuf: empty bytes.
- Server MUST always unmarshal the body unconditionally.

## Content Types

| Content-Type | Use |
|---|---|
| `application/json` | JSON requests and responses (UTF-8) |
| `application/protobuf` | Protobuf requests and responses |
| `application/octet-stream` | Unstructured binary data |

- Server MUST always support JSON as fallback.
- Server MUST always set `Content-Type` on responses.
- Server MUST include `X-DUH-Version` header on every response.
- No mime type parameters (no `;charset=utf-8`). No multiple mime types.
- If the server cannot satisfy the requested type, return `455` with a Reply.

## Reply Structure

Every non-200 service response uses this structure. It MAY also be used for simple 200 responses.

```json
{
  "code": "<string>",
  "message": "<string>",
  "details": { "<key>": "<value>" }
}
```

| Field | Required | Type | Description |
|---|---|---|---|
| `code` | Yes | string | Application-level code. May be numeric (`"400"`, `"453"`) or semantic (`"CARD_DECLINED"`) |
| `message` | No | string | Human-readable message |
| `details` | No | map\<string, string\> | Additional machine-readable context |

The HTTP status code is authoritative for retry and routing. The `code` field is an application-level signal and does not need to mirror the HTTP status.

## HTTP Status Codes

| Code | Meaning | Retry | Responsible |
|---|---|---|---|
| 200 | OK | N/A | Service |
| 400 | Bad Request | No | Service |
| 401 | Unauthorized (AuthN) | No | Auth |
| 403 | Forbidden (AuthZ) | No | Auth |
| 404 | Not Found | No | Service |
| 409 | Conflict | No | Service |
| 429 | Too Many Requests | Yes | Rate Limiter |
| 452 | Client Error (synthetic, never on wire) | No | Client SDK |
| 453 | Request Failed | No | Service |
| 454 | Retry Request | Yes | Service |
| 455 | Client Content Error | No | DUH-RPC framework |
| 500 | Internal Error | Yes | Service |
| 501 | Not Implemented | No | DUH-RPC framework |

When implementing a service endpoint, you will typically return: 200, 400, 404, 409, 453, 454, 500. Codes 455 and 501 are handled by the DUH-RPC framework. Codes 401, 403, and 429 are handled by auth/rate-limit middleware.

Do NOT invent new HTTP status codes. Use the `details` map for application-specific error codes.

## Infrastructure vs Service Errors

The server MUST include the `X-DUH-Version` header on every response. If the header is present, the response is from the service and the Reply body contains error details.

An infrastructure error is a response that does NOT include the `X-DUH-Version` header **and** has an HTTP status code of 5xx or 404. A missing header on any other status code (e.g. 200, 400, 401) indicates a non-conforming service, not an infrastructure error.

| Has `X-DUH-Version`? | HTTP code | Origin | Action |
|---|---|---|---|
| Yes | 404 | Service — resource not found | Do not retry |
| No | 404 | Infrastructure — service unreachable | Retry |
| Yes | 500 | Service — internal error | Retry |
| No | 502 | Infrastructure — gateway error | Retry |
| No | 400 | Non-conforming service | Do not retry |

The `X-DUH-Version` check is a last-resort signal. Evaluate the HTTP status code, content type, and response body first. Only check for the absence of `X-DUH-Version` on 5xx or 404 when no other rule applies.

## Client-Side Code 452

Synthetic code generated by the client SDK. Never sent over the wire. Never retry.

| Scenario | Description |
|---|---|
| Marshal failure | Request payload could not be serialized |
| Request construction failure | HTTP request object could not be created |
| Transport failure | `http.Client.Do()` failed (DNS, connection refused) |
| Response deserialization failure | Server returned 200 but body could not be unmarshalled |

## Retry Rules

Retry when:
1. HTTP status is marked Retry: Yes in the status code table above.
2. Response is an infrastructure error (no `X-DUH-Version` header **and** status is 5xx or 404).

Do NOT retry:
- 400, 401, 403, 404 (with `X-DUH-Version`), 409, 452, 453, 455, 501
- Any response missing `X-DUH-Version` on a non-5xx/non-404 status code

Always retry with backoff. Strategy and max duration are implementation-defined.

**Idempotency caveat:** Non-idempotent operations (`.create`, `.send`) risk duplication on retry after infrastructure errors. Services SHOULD provide an idempotency key if they want these safely retryable.

## Rate Limiting

A `429` with a Reply body is a service-level rate limit:

```json
{
  "code": "429",
  "message": "Rate limit exceeded",
  "details": {
    "ratelimit-limit": "1000",
    "ratelimit-remaining": "0",
    "ratelimit-reset": "30"
  }
}
```

| Detail key | Description |
|---|---|
| `ratelimit-limit` | Total requests allowed in window |
| `ratelimit-remaining` | Requests remaining |
| `ratelimit-reset` | Seconds until reset (supports decimals) |

Services SHOULD also include a `Retry-After` header (delay-seconds form).

A `429` without a Reply body is an infrastructure-level rate limit. Retry with backoff.

## Pagination

Cursor-based forward pagination only. No `limit`, `offset`, or `page` parameters.

**Request** — nest under `pagination`:
```json
{
  "pagination": {
    "first": 25,
    "after": "cursor_abc"
  }
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `pagination.first` | integer | Yes | Items to return (1-100) |
| `pagination.after` | string | No | Cursor from previous response |

**Response:**
```json
{
  "items": [...],
  "pagination": {
    "end_cursor": "cursor_xyz",
    "has_next_page": true
  }
}
```

| Field | Type | Required | Description |
|---|---|---|---|
| `items` | array | Yes | Page of results |
| `pagination.end_cursor` | string | Yes | Cursor for next page |
| `pagination.has_next_page` | boolean | Yes | More results exist |

Stop requesting when `has_next_page` is `false`.

An endpoint is paginated if its response has `items` + `pagination`. Actions like `.list` and `.search` that return collections SHOULD be paginated.

## Schema Rules

- Every operation MUST have a dedicated request schema and a dedicated response schema. No sharing across operations.
- Name schemas after their operation: `UserCreateRequest`, `UserCreateResponse`.
- No `readOnly` or `writeOnly` field annotations. Put fields in the correct schema instead.

### Protobuf Compatibility Constraints

These constraints ensure schemas can be represented as protobuf messages:

| Rule | Reason |
|---|---|
| No `nullable: true` fields | Protobuf has no null. Use field absence. |
| All integers/numbers MUST specify format (`int32`, `int64`, `float`, `double`) | Protobuf requires exact numeric types |
| No nested arrays (array of arrays) | Protobuf cannot represent `repeated repeated`. Wrap inner array in a message. |
| `additionalProperties` MUST have explicit value type | Protobuf maps need typed keys and values |
| No `allOf` | No protobuf equivalent |
| No `anyOf` | Ambiguous typing, no protobuf equivalent |
| `oneOf` only with discriminator + mapping | Bare `oneOf` is not permitted |

## Streaming

Three streaming content types:

| Content Type | Encoding | Use Case |
|---|---|---|
| `application/octet-stream` | Raw bytes | Unstructured binary transfer |
| `application/duh-stream+json` | JSON | Server-to-client structured stream |
| `application/duh-stream+protobuf` | Protobuf | Server-to-client structured stream |

All streaming endpoints use POST.

### Unstructured Streams (`application/octet-stream`)

- Errors MUST be returned as a Reply **before** any bytes are sent. Once bytes flow, there is no way to inject a Reply.
- A `200` with `Content-Type: application/octet-stream` means raw bytes — do NOT parse as Reply.
- Mid-stream disconnection is an infrastructure error. Client MAY retry from the beginning.
- Resumable transfers supported via standard HTTP `Range` / `Content-Range` headers (optional).

### Structured Streams (`duh-stream+json` / `duh-stream+protobuf`)

A sequence of typed messages over a single HTTP response. One payload type per stream.

**Wire format — length-prefix framing:**
```
[ 1 byte: flag ][ 4 bytes: uint32 big-endian length ][ N bytes: payload ]
```

**Frame types:**

| Flag | Meaning | Payload | Terminal |
|---|---|---|---|
| `0x0` | Data frame | Single instance of the stream's payload type | No |
| `0x1` | Final frame | Same payload type (MAY be empty, length=0) | Yes |
| `0x2` | Error frame | Reply structure | Yes |

Rules:
- After a final or error frame, the stream is closed. No further frames.
- The client MUST be prepared to receive an error frame at any point, not just the start.
- `Content-Type` on response echoes the client's `Accept` header.
- The JSON fallback rule does **not** apply to streaming. If the server cannot satisfy the requested stream content type, return `400` with a Reply. No silent fallback.
- Mid-stream disconnection before final/error frame is an infrastructure error. Client MAY retry from the beginning.

**Resumption:** Not built into the framing layer. To support resumption, encode a `sequence` field in your payload type. On reconnect, the client sends the last received sequence in the request body.

### Browser Clients

Do NOT use SSE / `EventSource` (GET-only, no request body). Use `application/duh-stream+json` via `fetch()`.

### Bidirectional Streams

Not yet defined.
